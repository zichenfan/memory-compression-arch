# 混合粒度内存压缩架构项目规划
## 1. 项目背景与挑战
当前内存压缩技术面临"压缩率"与"解压延迟"的根本矛盾：
- **大粒度 (如 4KB Page) + 复杂算法 (如 ZSTD, LZ4)**：
  - 优势：能找到更多重复模式，压缩率高（冷数据友好）。
  - 劣势：解压路径长，随机访问小块数据时存在严重的读放大（Read Amplification）和延迟，无法满足热数据的严格时延要求。
- **小粒度 (如 64B/256B Cache Line) + 简单算法 (如 FPC, BDI, BPC)**：
  - 优势：硬件实现简单，单周期或少周期即可完成解压（热数据友好）。
  - 劣势：字典/模式范围太小，整体压缩率差，无法显著扩大有效内存容量。

## 2. 核心解决思路：冷热自适应混合压缩架构
根据数据的访问温度（Temperature），动态选择最合适的压缩策略：
- **冷数据（低频访问）**：采用 4KB 粒度 + 强力字典压缩（ZSTD 变种 / LZ-based），追求最大化容量。
- **温热数据（高频访问）**：采用 256B 粒度 + 模式匹配压缩（FPC / BDI），追求极致解压速度。

### 关键创新点：硬件资源复用 (Unified Decompression Engine)
为了避免硬件面积（Area）的线性增加，我们需要设计一套**统一的硬件加速引擎**，能够兼顾两种截然不同的解压范式。
- **发现共性**：复杂算法（ZSTD/LZ）的核心操作是**字典匹配与复制**（Literal copy & Match copy）；简单算法（FPC）的核心是**零游程编码（RLE）和短模式匹配**（如全零、符号扩展）。
- **架构融合**：将 FPC 的模式匹配器（Pattern Matcher）作为 LZ 字典解压器的**特例或子集**。在解压 4KB 数据时，引擎全流水线运行；在解压 256B 时，旁路（Bypass）掉长字典查找表，仅激活单周期的模式展开模块。

## 3. 详细架构分析与技术路线
### 3.1 算法特征对齐分析

| 特征 | 方案 A (如 4KB + LZ/ZSTD) | 方案 B (如 256B + FPC) | 统一硬件复用思路 |
| --- | --- | --- | --- |
| **数据单位** | 4096 Bytes | 256 Bytes | 设计支持可变步长(256B为最小块)的流水线 |
| **字典机制** | 动态历史窗口 (如2KB-4KB) | 静态/极短模式 (如连续0/扩展位) | 使用多体SRAM，小块时关闭长历史区降功耗 |
| **并行度** | 通常较难并行 (存在数据依赖) | 极高并行度 (各word独立) | 构建多个小的处理单元(PE)，大块时级联，小块时并行 |
| **状态机** | 复杂的 Huffman/FSE 解码状态机 | 简单的 3-bit 前缀选择器 | FSE解码器与前缀解码器复用取指单元 |

### 3.2 硬件微架构设计 (Unified Decompression Microarchitecture)
我们将设计一个包含以下模块的加速器：
1. **多模取指与解码前端 (Multi-mode Fetch & Decode)**
  - 根据块的元数据（Header）判断压缩类型。
  - 对于热数据，激活快速前缀解码器；对于冷数据，激活 FSE/Huffman 解码器。
2. **可重构处理单元阵列 (Reconfigurable PE Array)**
  - 将数据路径划分为多个 64-Byte 的 PE。
  - **小粒度模式**：4个 PE 并行处理 256B 的 FPC 解压。
  - **大粒度模式**：PE 形成流水线环，配合历史字典进行长数据的 Match Copy。
3. **混合历史缓存 (Hybrid History Buffer)**
  - 作为 ZSTD 的滑动窗口。在 FPC 模式下，可以复用为解压后数据的临时写回缓冲区（Write-back Buffer），掩盖 DRAM 写入延迟。

### 3.3 系统级集成 (OS & Memory Controller)
- **温控机制**：在内存控制器（Memory Controller）或 OS 页表（Page Table）中维护访问计数器。
- **数据迁移与重压缩**：
  - 热转冷：后台硬件引擎或 OS 守护进程将多个 256B 的热块收集，重压缩为 4KB 的冷块。
  - 冷转热：当 4KB 冷块被高频访问时，解压后将其以 256B 粒度重新压缩存放。
- **地址映射**：需要一个轻量级的压缩元数据表（Compression Metadata Table），支持变长块管理。

## 4. 科研项目管理计划 (Phases)
### Phase 1: 算法建模与软件仿真 (Month 1-2)
- 收集典型的内存负载 Trace (SPEC CPU, 数据库等)。
- 使用 Python/C++ 编写软件模型，对比不同粒度（256B vs 4KB）和算法组合的压缩率和理论访存次数。
- 确定最优的"冷热阈值"和具体的算法参数（例如 ZSTD 的窗口大小、FPC 的具体模式集）。

### Phase 2: 统一硬件微架构设计 (Month 3-4)
- 详细设计可重构数据路径 (Verilog/Chisel 级规划)。
- 使用架构级模拟器 (如 gem5 或 NVMain) 评估该引擎在真实系统下的端到端性能 (IPC 提升、内存带宽节省)。
- **里程碑：输出微架构设计文档和模拟器性能数据。**

### Phase 3: RTL 实现与综合评估 (Month 5-6)
- 编写核心加速器的 RTL 代码。
- 进行逻辑综合，获取准确的面积（Area）、功耗（Power）和时序（Timing）。
- 验证复用逻辑带来的面积收益，与分别做两个独立加速器的 Baseline 进行对比。

### Phase 4: 论文撰写与开源 (Month 7-8)
- 整理实验数据，绘制图表。
- 将硬件代码、软件模型和 gem5 补丁推送到 GitHub 仓库。
- 撰写顶会论文 (如 ISCA, MICRO, HPCA 级别)。
